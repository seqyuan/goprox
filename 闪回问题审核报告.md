# GoProx 闪回问题全面审核报告

## 📋 执行摘要

本次审核针对 GoProx 项目的**闪回问题**（页面刷新/点击后退回登录页或其他页面的问题）进行了全面分析。涉及多个端口应用、点击和刷新功能。

**问题级别：🔴 严重**  
**影响范围：** 前端卡片点击、应用内刷新、浏览器后退/前进、WebSocket连接、SPA应用路由

## 🎯 发现的关键问题

### 1. 【严重】Session Cookie 缺少刷新机制
**位置：** `internal/auth/auth.go`

**问题描述：**
- Session cookie 使用固定 TTL（默认 86400 秒），但缺少滑动窗口刷新机制
- 用户活跃使用期间，session 仍然会过期，导致突然"闪回"到登录页
- 只在登录时设置一次 cookie，后续请求不更新过期时间

**影响：**
```
用户场景：上午 9:00 登录 → 下午 9:00 (24小时后) 点击任何链接 → 闪回登录页
```

**建议修复：**
```go
// 在 server.go 的每个需要认证的请求中添加：
if session.Valid && session.UserID != "" {
    // 距离过期还有不到 25% 时间时刷新
    if shouldRefreshSession(session, state.Auth.SessionTTL) {
        cookie := auth.SetSessionCookie(s.sessionSecret, s.state.Auth.SessionTTL, 
                                        session.UserID, auth.IsSecureRequest(r))
        w.Header().Add("Set-Cookie", cookie)
    }
}
```

### 2. 【严重】Route Cookie 路径不正确导致后续请求失效
**位置：** `internal/auth/auth.go:151` + `internal/server/server.go:139,165`

**问题描述：**
```go
// 当前实现
func SetRouteCookie(prefix string) string {
    return fmt.Sprintf("%s=%s; HttpOnly; Path=/; SameSite=Lax",
        RouteCookieName, prefix)
}
```

**存在的问题：**
1. `Path=/` 使得 cookie 对所有路径有效，但 cookie 值存储的是特定服务前缀
2. 当用户在不同服务间切换时，route cookie 会被覆盖
3. 用户点击 A 服务 → route cookie = "/proxy/user/jupyter/" → 在 A 服务内点击链接 `/api/xxx` → 系统用 jupyter 的 route cookie 匹配 → 可能失败

**影响场景：**
- 在 Jupyter 内点击"新建笔记本" → 可能闪回登录页
- 在 RStudio 内刷新页面 → 可能闪回登录页
- 多标签页切换不同服务 → route cookie 混乱

**建议修复方案 A（推荐）：**
```go
// 改为作用域限定的 cookie
func SetRouteCookie(prefix string) string {
    // Cookie 只在该服务路径下有效
    return fmt.Sprintf("%s=%s; HttpOnly; Path=%s; SameSite=Lax",
        RouteCookieName, prefix, prefix)
}
```

**建议修复方案 B：**
```go
// 使用 sessionStorage 替代 cookie（需前端配合）
// 或使用 URL 参数传递上下文
```

### 3. 【严重】Referer 头依赖不可靠
**位置：** `internal/server/server.go:239-266`

**问题描述：**
```go
func (s *Server) handleRefererProxy(w http.ResponseWriter, r *http.Request) bool {
    referer := r.Header.Get("Referer")
    if referer == "" {
        return false
    }
    // ...
}
```

**缺陷：**
1. 浏览器刷新（F5）时，Referer 是当前页面，可正常工作
2. 新标签页打开链接 → Referer 为空 → 无法匹配 → 闪回登录页
3. 浏览器后退/前进 → Referer 可能丢失 → 闪回登录页
4. HTTPS → HTTP 降级 → 浏览器不发送 Referer → 闪回登录页
5. 某些浏览器隐私模式/插件会屏蔽 Referer

**影响场景：**
- 用户在 Jupyter 内右键"在新标签页打开" → 闪回登录页
- 用户点击浏览器后退按钮 → 闪回登录页
- 使用隐私浏览器插件 → 所有请求闪回

**建议修复：**
1. 不应该把 Referer 作为主要路由机制
2. 使用更可靠的状态传递：
   - URL 路径本身（首选）
   - Session-scoped storage
   - 加密的 state token

### 4. 【高】WebSocket 升级缺少验证和超时处理
**位置：** `internal/proxy/proxy.go` （未实现 WebSocket 特殊处理）

**问题描述：**
- 代码使用 `httputil.ReverseProxy`，它会自动处理 WebSocket 升级
- 但缺少以下关键检查：
  1. 没有验证 `ServiceConfig.WebSocket` 标志
  2. 没有对不支持 WebSocket 的服务拒绝升级请求
  3. 没有 WebSocket 连接超时处理

**影响：**
- 配置中 `websocket: false` 的服务仍然可以建立 WebSocket 连接
- WebSocket 连接可能无限期挂起，耗尽服务器资源
- 后端服务重启时，WebSocket 连接不会优雅重连，用户看到断线

**建议修复：**
```go
// 在 proxy.go NewReverseProxy 中添加
func NewReverseProxy(svc *config.ServiceConfig, fc ProxyForwardContext) *httputil.ReverseProxy {
    target := &url.URL{
        Scheme: "http",
        Host:   svc.Host + ":" + strconv.Itoa(svc.Port),
    }

    director := func(req *http.Request) {
        // WebSocket 升级检查
        if isWebSocketUpgrade(req) {
            if !svc.WebSocket {
                // 应该在这里拦截，但 Director 不能返回错误
                // 需要在外层 ServeHTTP 前检查
            }
        }
        // ... 原有逻辑
    }
    
    // ...
}

// 在 server.go 的 forwardToBackend 前添加
func (s *Server) forwardToBackend(...) {
    if isWebSocketUpgrade(r) && !svc.WebSocket {
        http.Error(w, "WebSocket not enabled for this service", http.StatusForbidden)
        return
    }
    // ...
}

func isWebSocketUpgrade(r *http.Request) bool {
    return strings.ToLower(r.Header.Get("Connection")) == "upgrade" &&
           strings.ToLower(r.Header.Get("Upgrade")) == "websocket"
}
```

### 5. 【高】前端卡片点击后缺少加载状态反馈
**位置：** `internal/web/templates/dashboard.html` + `script.js`

**问题描述：**
```html
<a class="card-link" href="..." target="_blank">
```

**缺陷：**
1. 点击卡片后立即在新标签页打开，但用户看不到加载状态
2. 如果后端服务启动慢（如 Jupyter 需要 5-10 秒启动），用户会看到空白页或错误
3. 新标签页可能因为 session 检查失败而闪回登录页

**影响场景：**
- 点击 Jupyter 卡片 → 新标签页空白 5 秒 → 超时 → 闪回 404
- 点击 RStudio 卡片 → 显示 502 Bad Gateway → 用户以为服务挂了

**建议修复：**
```javascript
// 修改卡片点击处理
$$('.card-link').forEach(function(link) {
  link.addEventListener('click', function(ev) {
    ev.preventDefault();
    var url = link.getAttribute('href');
    
    // 显示加载提示
    showToast('正在连接服务...');
    
    // 先健康检查
    fetch(url + '?_health=1', { credentials: 'same-origin' })
      .then(function(res) {
        if (res.ok) {
          window.open(url, '_blank');
        } else {
          showToast('服务未就绪，请稍后再试');
        }
      })
      .catch(function() {
        // 即使健康检查失败也打开，让用户看到具体错误
        window.open(url, '_blank');
      });
  });
});
```

### 6. 【中】Dashboard 自动刷新会打断用户操作
**位置：** `internal/web/templates/script.js:226-234`

**问题描述：**
```javascript
jsonFetch("/api/services").then(function(data) {
    state.services = data.services || state.services;
    state.writable = !!data.writable;
    // ...
    renderDashboard();  // 直接重新渲染
});
```

**缺陷：**
1. 页面加载时会调用一次 `/api/services`
2. 如果后端 `ScanLoop` 检测到配置变化，前端不会自动知道
3. 代码没有轮询机制，但如果未来添加轮询，`renderDashboard()` 会：
   - 清空当前拖拽状态
   - 关闭正在编辑的模态框
   - 丢失用户输入

**建议修复：**
```javascript
// 添加智能刷新逻辑
function smartRefresh(newServices) {
  // 只在服务列表真正变化时才刷新
  if (JSON.stringify(newServices) === JSON.stringify(state.services)) {
    return;
  }
  
  // 检查是否有活跃操作
  var hasActiveModal = $('#add-modal').classList.contains('open') ||
                       $('#edit-modal').classList.contains('open');
  var isDragging = dragId !== null;
  
  if (hasActiveModal || isDragging) {
    // 延迟刷新，设置标记
    pendingRefresh = newServices;
    return;
  }
  
  // 安全刷新
  state.services = newServices;
  renderDashboard();
}
```

### 7. 【中】API 端点路由冲突导致后端服务 /api 请求失败
**位置：** `internal/server/server.go:42-52`

**问题描述：**
```go
mux.HandleFunc("/api/", func(w http.ResponseWriter, r *http.Request) {
    if s.apiHandler.ServeHTTP(w, r) {
        return
    }
    // 尝试转发到后端服务
    if s.handleRouteCookieProxy(w, r) {
        return
    }
    if s.handleRefererProxy(w, r) {
        return
    }
    http.NotFound(w, r)
})
```

**缺陷：**
1. GoProx 的 API 路由：`/api/services`, `/api/services/layout` 等
2. 如果用户的 SPA（如 Next.js/AnnoVibe）也使用 `/api/*` 路径
3. 冲突处理逻辑依赖 `handleRouteCookieProxy` 和 `handleRefererProxy`
4. 这两个方法不可靠（见问题 2、3），导致：
   - 用户在 AnnoVibe 内调用 `/api/data` → 匹配不到后端 → 404
   - 用户点击刷新 → Referer 丢失 → 闪回 404

**影响场景：**
- 任何使用 `/api` 路径的 SPA 应用都会断断续续失败
- Next.js 的 API Routes 无法使用
- Vue/React SPA 的 API 调用随机失败

**建议修复方案 A（推荐）：**
```go
// 改用更具体的前缀，避免冲突
mux.HandleFunc("/goprox-api/", s.apiHandler.ServeHTTP)

// 前端相应修改
jsonFetch("/goprox-api/services")
```

**建议修复方案 B：**
```go
// 严格匹配 GoProx API 路径
mux.HandleFunc("/api/services", s.handleGoProxAPI)
mux.HandleFunc("/api/services/", s.handleGoProxAPI)
mux.HandleFunc("/api/services/layout", s.handleGoProxAPI)

// 其他 /api/* 全部转发到后端
mux.HandleFunc("/api/", s.handleAPIProxy)
```

### 8. 【中】后端登录表单检测逻辑脆弱
**位置：** `internal/server/server.go:131-155`

**问题描述：**
```go
bodyBytes, err := io.ReadAll(r.Body)
// ...
if strings.Contains(string(bodyBytes), "username=") {
    // 这是 GoProx 登录
    r.Body = io.NopCloser(strings.NewReader(string(bodyBytes)))
    return false
}
```

**缺陷：**
1. 检测方式过于简单：只检查是否包含 `username=` 字符串
2. 如果后端应用（如自定义 web 服务）的登录表单也使用 `username` 字段 → 误判
3. 如果 GoProx 用户名恰好包含 `username=` 字符串 → 误判
4. 整个 body 读取到内存，大请求会有性能问题

**影响场景：**
- 用户自己开发的 web 服务使用 `username/password` 登录 → 被拦截 → 无法登录后端
- 用户提交大表单（如上传文件） → 内存占用高

**建议修复：**
```go
func (s *Server) handleBackendLogin(w http.ResponseWriter, r *http.Request) bool {
    session := auth.GetSessionFromCookies(r.Header.Get("Cookie"), s.sessionSecret)
    if !session.Valid || session.UserID == "" {
        return false
    }

    match := s.findBackendService(r, session.UserID)
    if match == nil {
        return false
    }

    // 更可靠的检测：检查 Content-Type 和 Origin
    ct := r.Header.Get("Content-Type")
    origin := r.Header.Get("Origin")
    referer := r.Header.Get("Referer")
    
    // GoProx 登录特征：
    // 1. Origin/Referer 是网关根路径
    // 2. Content-Type 是表单
    // 3. 路径是 /login (不是 /proxy/.../login)
    isGoProxLogin := r.URL.Path == "/login" &&
                     strings.HasPrefix(ct, "application/x-www-form-urlencoded") &&
                     (origin == "" || !strings.Contains(origin, "/proxy/")) &&
                     (referer == "" || !strings.Contains(referer, "/proxy/"))
    
    if isGoProxLogin {
        return false  // 让 GoProx 处理
    }

    // 其他情况转发到后端
    s.forwardAsIs(w, r, match)
    return true
}
```

## 🛠️ 推荐的架构改进

### 1. 引入显式路由上下文传递
```
当前：依赖 Cookie + Referer 隐式推断
建议：URL 中嵌入加密的上下文 token

/proxy/{user}/{service}/xxx
    ↓ 改为
/proxy/{user}/{service}/xxx?_ctx={encrypted_context}

好处：
- 不依赖 Cookie 和 Referer
- 支持直接分享链接
- 跨标签页一致性
```

### 2. 实现健康检查端点
```go
// 添加到 server.go
mux.HandleFunc("/proxy/{user}/{service}/_health", func(w http.ResponseWriter, r *http.Request) {
    // 检查后端服务是否就绪
    match := s.registry.FindService(r.URL.Path)
    if match == nil {
        w.WriteHeader(503)
        return
    }
    
    // 尝试连接后端
    conn, err := net.DialTimeout("tcp", 
        net.JoinHostPort(match.Service.Host, strconv.Itoa(match.Service.Port)),
        2*time.Second)
    if err != nil {
        w.WriteHeader(503)
        json.NewEncoder(w).Encode(map[string]string{"status": "unavailable"})
        return
    }
    conn.Close()
    
    w.WriteHeader(200)
    json.NewEncoder(w).Encode(map[string]string{"status": "ok"})
})
```

### 3. 添加前端状态持久化
```javascript
// 使用 sessionStorage 记住用户当前服务上下文
function updateContext(servicePath) {
  sessionStorage.setItem('goprox_current_service', servicePath);
}

function getCurrentContext() {
  return sessionStorage.getItem('goprox_current_service') || '';
}

// 在所有 API 请求中携带上下文
function jsonFetch(path, opts) {
  var ctx = getCurrentContext();
  if (ctx) {
    opts = opts || {};
    opts.headers = opts.headers || {};
    opts.headers['X-GoProx-Context'] = ctx;
  }
  return fetch(path, opts);
}
```

## ✅ 总结

GoProx 项目的闪回问题主要由以下原因导致：

1. **Session 管理不完善**：固定过期时间，缺少活跃刷新
2. **路由上下文丢失**：过度依赖不可靠的 Cookie 和 Referer
3. **多服务环境下状态冲突**：单一 route cookie 被覆盖

**核心修复策略：**
- 实现 Session 滑动窗口
- 重构路由匹配逻辑，减少对 Referer 的依赖
- 使用路径范围限定的 Cookie
- 添加显式的上下文传递机制

**预期效果：**
修复后，用户在使用过程中应该：
- ✅ 长时间活跃不会被踢出
- ✅ 新标签页打开链接正常工作
